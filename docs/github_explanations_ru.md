# Интерактивное объяснение экстрактора Zombie Infection

Этот документ предназначен для чтения прямо на GitHub: через оглавление, якорные ссылки и короткие блоки «что это даёт». Он объясняет, зачем нужен экстрактор, как устроен пайплайн и какую роль играет каждая основная часть кода.

> Важно по точке входа:
>
> - `offline_extractor.py` — **тонкий backward-compatible wrapper** для старых сценариев запуска;
> - «истинный» основной пайплайн и CLI находятся в `tools/extract_zombie_infection.py` (`run_extractor` и `main`).

## Что делает экстрактор

`offline_extractor.py` сохранён для совместимости, но основная логика экстракции теперь живёт в `tools/extract_zombie_infection.py`.

Основной оркестратор не просто копирует данные из архива, а пытается:

1. распознать структуру контейнеров `t0`, `m0..m13_*`;
2. отделить заголовки от полезных данных;
3. извлечь текст, карты, скрипты, аудио и исследовательские превью графики;
4. сохранить результаты в удобном для анализа виде (`.json`, `.txt`, `.png`, `.mid`);
5. собрать человекочитаемые сводки по найденным данным.

## Как проходит обработка JAR

Полный путь обработки в `run_extractor()` выглядит так:

1. `parse_packs()`
2. `decode_text()`
3. `decode_audio()`
4. `decode_maps()`
5. `decode_graphics()`
6. `extract_ui_assets()`
7. `build_final_table()`
8. `build_chapter_mission_matrix()`
9. `build_chapter_matrix()`
10. Формируется единый `summary.json`, который связывает все найденные артефакты.

## Ключевые структуры и функции

### `Container`

Это базовый уровень разбора бинарных контейнеров.

Что делает `Container`:

- определяет, используется ли заголовок с `u32-count` или `u8-count`;
- читает таблицу смещений;
- вычисляет `header_size`;
- строит `ranges` для каждого чанка;
- сохраняет уже обрезанные `payloads` без служебной части заголовка.

**Зачем это нужно:** дальше вся логика экстрактора работает уже не с сырыми архивными блоками, а с нормализованными чанками, которые проще анализировать и сравнивать.

### `export_chunks`

Функция выгружает каждый чанк контейнера как отдельный `.bin` и одновременно создаёт структурное описание чанков в памяти.

Что сохраняется дополнительно:

- индекс чанка;
- границы в контейнере;
- размер;
- CRC32;
- классификация (`string_table`, `midi`, `audio_blob`, `tile_or_collision_grid` и т.д.).

**Зачем это нужно:** даже если конкретный формат чанка пока неизвестен, у исследователя уже есть каталог всех элементов и способ быстро сверять одинаковые или похожие чанки по CRC и размеру.

### `decode_game_text` и `export_strings`

Этот блок отвечает за текстовые ресурсы `t0`.

#### `decode_game_text`

Пытается восстановить русский текст, если он прошёл через неправильную перекодировку UTF-8 / CP1251.

#### `sanitize_text`

Убирает управляющие символы и бинарный мусор, чтобы текст было удобно читать.

#### `guess_offset_table`

Пытается угадать таблицу смещений внутри чанка и тем самым разбить большой бинарный блок на отдельные строки.

#### `export_strings`

Создаёт:

- полный текстовый дамп чанка;
- `*_segments_guess.json` с гипотезой о разбиении на строки;
- `*_reconstructed.txt` с более читаемой реконструкцией текста.

**Зачем это нужно:** для реверса UI, диалогов и названий локаций важен не просто бинарный дамп, а максимально читаемый текстовый слой.

### `export_audio`

Работает с `m13_1` и `m13_2`.

Логика простая:

- если в чанке найден `MThd`, он считается MIDI и сохраняется как `.mid`;
- если MIDI не найден, blob сохраняется как бинарный файл плюс рядом создаётся JSON-описание через `analyse_audio_blob()`.

JSON-описание содержит, например:

- размер;
- начальные байты;
- количество ненулевых байт;
- самые частые значения байтов.

**Зачем это нужно:** так можно отделить уже полезные музыкальные ресурсы от пока нерасшифрованных звуковых blob-ов и не потерять информацию о них.

### `parse_script_stream` и `export_script_and_level_packs`

Этот блок занимается сценарными и логическими данными.

#### `parse_script_stream`

Пытается пройти по бинарному потоку как по последовательности команд:

- читает `opcode`;
- собирает гистограмму команд;
- достаёт параметры;
- отдельно отмечает возможные ссылки на чанки `m8` и специальные команды.

#### `export_script_and_level_packs`

Применяет этот парсер к:

- `m8`;
- скриптовым частям `m9`;
- метаданным `m10`.

На выходе получаются JSON-файлы, которые уже можно читать на GitHub без просмотра hex-дампов.

**Зачем это нужно:** это первая ступень к пониманию структуры миссий, событий, триггеров и сценарных переходов.

### `export_tile_maps`

Этот блок анализирует `m6_0..m6_5` как карты/сетки тайлов.

Что происходит:

- данные читаются как массив `u16`;
- через `factor_grid()` подбирается предполагаемая ширина и высота;
- создаётся цветное превью через `pseudo_color()`;
- отдельно строится вариант как будто данные кодируют `RGB565`;
- сохраняется JSON с размерами, диапазоном значений и sidecar-данными.

**Зачем это нужно:** даже без полного знания формата можно быстро увидеть, похож ли чанк на карту, тайловую сетку или другой двумерный ресурс.

### `export_graphics_research`

Предназначен для исследовательского просмотра графических паков `m3_0`, `m4_0`, `m11_0`, `m11_1`.

Делает два вида превью:

- grayscale;
- RGB565.

Результат не считается «финальным декодером спрайтов», но помогает визуально понять:

- где повторяющиеся блоки;
- где возможно плиточное хранение;
- где могут быть данные палитры/атласа.

**Зачем это нужно:** это быстрый визуальный инструмент для реверса графики без написания полноценного декодера под каждый формат.

### `build_final_table`

Создаёт итоговую сводную таблицу по главам и миссиям.

Туда попадают:

- глава;
- название/описание миссии;
- скриптовый пак;
- пак карт;
- пак графики;
- аудио-подсказки;
- враги;
- ключевые сюжетные события.

**Зачем это нужно:** чтобы результаты реверса можно было быстро читать не только как технические данные, но и как игровую структуру.

### `write_research_notes`

Формирует markdown-заметки для исследователя:

- что уже реализовано;
- какие пакеты что содержат;
- сколько найдено MIDI;
- сколько карт найдено;
- какие строки особенно полезны как доказательство гипотез.

**Зачем это нужно:** удобно смотреть историю прогресса реверса прямо на GitHub, без запуска кода.

## Какие артефакты появляются на выходе

Обычно экстрактор создаёт такие группы файлов:

- `chunks/` — сырые чанки контейнеров;
- `extracted/text/` — текстовые дампы и реконструкции;
- `extracted/audio/` — MIDI и описания бинарного аудио;
- `extracted/maps/` — JSON и PNG по картам;
- `extracted/images/research/` — исследовательские превью графики;
- `docs/reverse_engineering/` — JSON/markdown-заметки и итоговая таблица;
- `summary.json` — общая карта результатов.

### `summary.json` как API для аналитики

`summary.json` закреплён как versioned API-контракт для консьюмеров данных (аналитика/дашборды/quality checks), а не как «временный внутренний формат».

Базовые правила чтения:

1. Сначала проверять `summary_schema_version` (semver).
2. Проверять поддерживаемый `MAJOR` (для текущего контракта: `1.x`).
3. Требовать обязательные ключи и игнорировать неизвестные дополнительные поля.

Минимальный пример чтения:

```python
import json
from pathlib import Path

summary = json.loads(Path(".artifacts/extractor_out/summary.json").read_text(encoding="utf-8"))
version = summary["summary_schema_version"]  # например: "1.0.0"
major = int(version.split(".", 1)[0])
if major != 1:
    raise RuntimeError(f"Неподдерживаемая major-версия схемы: {version}")

metrics = {
    "audio_coverage_percent": summary["audio_coverage"]["coverage_percent"],
    "maps_validation_failed": summary["maps_validation_failed"],
    "chapter_rows": summary["chapter_matrix_rows"],
}
print(metrics)
```

Полный список обязательных полей, инварианты и политика breaking/deprecation:
[`docs/reverse_engineering/summary_schema.md`](reverse_engineering/summary_schema.md).

## Почему результаты теперь не коммитятся

Теперь результаты экстрактора пишутся по умолчанию в `.artifacts/extractor_out/`, а сама папка добавлена в `.gitignore`.

Это сделано специально, чтобы:

- PR оставались обзорными и читабельными;
- в diff попадал только код и документация;
- бинарные файлы можно было публиковать отдельно как GitHub Artifacts, Releases или локальные артефакты исследования.

## Как читать проект на GitHub

Если открывать репозиторий на GitHub, рекомендуемый порядок такой:

1. открыть `README.md` — для общего понимания политики репозитория;
2. открыть этот документ — для навигации по архитектуре экстрактора;
3. открыть `offline_extractor.py` и читать по разделам, переходя по названиям функций из этого документа;
4. при необходимости запускать экстрактор локально и публиковать результаты отдельно как артефакты.

Если потребуется, следующий шаг — вынести эти объяснения ещё и в GitHub Wiki или в отдельный `docs/`-раздел с диаграммами пайплайна.
